סליקה פנימית ומנוע חשבוניות

מדריך יישום טכני להוספת שכבת מסחר ל‑WHITEBOARD: כל מרצה סולק את התלמידים שלו, הכסף עובר ישירות אליו, אנחנו מנפיקים חשבוניות מס בעצמנו (מנוע פנימי מול רשות המסים) וגוזרים נתח מעמלת הסליקה — בלי להחזיק כסף ובלי רישיון.

שלוש החלטות המוצר

סליקה: מסוף לכל מרצה

כל מרצה = עוסק עצמאי עם מסוף משלו. הכסף מסולק ישירות לחשבונו. אנחנו לא בזרימת הכסף → לא מאגד, אין רישיון.

חשבוניות: מנוע פנימי

אנחנו מתחברים ישירות ל‑Open API של רשות המסים ומנפיקים חשבונית מס בשם המרצה — בלי ספק חיצוני.

הכנסה: נתח מהעמלה

חברת הסליקה גובה מהמרצה ומעבירה לנו נתח (rev‑share). הכנסה נטו, אפס סיכון גבייה.

ספק סליקה מומלץ: PayPlus — תוכנית שותפים מתועדת (נתח 2%–10% + בונוס ₪100–195 למרצה), כל מרצה פותח חשבון משלו, השותף מחוץ לזרימת הכסף. HYP/PayMe כחלופות במו"מ. הארכיטקטורה למטה אגנוסטית לספק (Adapter).

ארכיטקטורה

ארבעה מודולים עצמאיים, כולם תחת ה‑LMS הקיים. כל אינטגרציה חיצונית עוברת דרך Adapter כדי שנוכל להחליף ספק סליקה בלי לגעת בלוגיקה.

Checkout

מודול סליקה

הטמעת דף תשלום (iframe) של מסוף המרצה, יצירת עסקה, קבלת Webhook על תשלום מוצלח. PaymentAdapter: PayPlus / HYP / PayMe.

Invoicing

מנוע חשבוניות

חיבור OAuth פר‑מרצה ל‑Open API של רשות המסים, משיכת מספר הקצאה, יצירת מסמך חשבונית חתום, ארכוב.

Revenue

התאמת עמלות

רישום מכירות פר‑מרצה (partner_id), התאמה מול דוח ה‑rev‑share של הספק, הנפקת חשבונית עמלה לספק.

Onboarding

אונבורדינג מרצה

אשף: פתיחת מסוף, אימות עוסק מורשה, מתן הרשאה דיגיטלית לרשות המסים, חתימה על הסכם שירות.

עקרון מנחה: WHITEBOARD לעולם לא מחזיקה בכספי הקורסים. כל זרימת כסף של מכירה היא תלמיד→מרצה. השרת שלנו רואה רק אירועים (Webhooks) ומנפיק מסמכים — לא כספים.

זרימת רכישה

1
התלמיד לוחץ "קנה קורס" → אנחנו פותחים את דף התשלום (iframe) של מסוף המרצה.
2
התלמיד משלם → הכסף מסולק ישירות לחשבון המרצה (אנחנו לא בזרימה).
3
הספק שולח Webhook מאומת (חתימת HMAC) לשרת שלנו: עסקה, סכום, אמצעי תשלום, מזהה מרצה.
4
אנחנו רושמים Sale, פותחים גישה לקורס לתלמיד, ומפעילים job להנפקת חשבונית.
5
job התאמת העמלות מצרף את המכירה לחישוב ה‑rev‑share החודשי מול הספק.
// app/api/webhooks/payment/route.ts — אימות וטיפול ב‑Webhook
export async function POST(req: Request) {
  const raw = await req.text();
  const sig = req.headers.get("x-signature");
  if (!paymentAdapter.verify(raw, sig)) return json({ error: "bad signature" }, 401);

  const evt = saleEventSchema.parse(JSON.parse(raw)); // Zod
  await prisma.sale.create({ data: { ...evt, status: "PAID" } });
  await grantCourseAccess(evt.studentId, evt.courseId);
  await tasks.trigger("issue-invoice", { saleId: evt.id }); // Trigger.dev
  return json({ ok: true });
}

זרימת הנפקת חשבונית

1
job issue-invoice נטען עם ה‑Sale. שולף את הטוקן התקף של המרצה מול רשות המסים.
2
אם העסקה B2B מעל הסף → מושך מספר הקצאה מ‑Open API. אם B2C → מדלג (לא נדרש).
3
בונה מסמך חשבונית מס: פרטי המרצה כמוכר, מספור רציף פר‑מרצה, מספר הקצאה (אם יש), חתימה דיגיטלית.
4
שומר ב‑DB, מארכב (Bunny), שולח לתלמיד במייל (Trigger.dev), ורושם ב‑קובץ אחיד (BKMVDATA).
בקרת §50 (קריטי): לפני הנפקה — לוודא שהמרצה הוא עוסק מורשה. עוסק פטור / לא‑רשום → לחסום הנפקת חשבונית מס (מותר רק קבלה/חשבונית עסקה). הנפקה שגויה = קנס כפל מס.

אונבורדינג מרצה

1
המרצה פותח מסוף אצל ספק הסליקה (דרך לינק שותף ייחודי שלנו — מיוחס ל‑partner_id).
2
אימות סטטוס: עוסק מורשה פעיל + הזנת מספר עוסק/ח.פ. (חוסם עוסק פטור).
3
הרשאה דיגיטלית: המרצה מופנה לאתר רשות המסים, מאשר את האפליקציה שלנו (OAuth), חוזר עם code.
4
חתימה על הסכם שירות (מנפיקים מסמכים בשמו; אחריות המס עליו; שיפוי). שמירת טוקן + refresh.
5
המרצה פעיל — יכול לתמחר קורסים ולמכור. סטטוס מוצג בדשבורד.

// trigger/renew-tax-tokens.ts — חידוש טוקני רשות המסים (~3 חודשים)
export const renewTaxTokens = schedules.task({
  id: "renew-tax-tokens",
  cron: "0 3 * * *",
  run: async () => {
    const soon = await prisma.taxAuthGrant.findMany({
      where: { expiresAt: { lte: addDays(new Date(), 10) } },
    });
    for (const g of soon) {
      const next = await taxApi.refresh(g.refreshToken).catch(() => null);
      if (!next) { await notifyLecturer(g.lecturerId, "renew_auth"); continue; }
      await prisma.taxAuthGrant.update({ where: { id: g.id }, data: next });
    }
  },
});

ספק הסליקה — Adapter

ממשק אחיד שמסתיר את הספק. מימוש ראשון: PayPlus. כל ספק חדש = מחלקה שמממשת את ה‑interface.

interface PaymentAdapter {
  createCheckout(p: { lecturerId: string; amount: number; courseId: string }): Promise<{ iframeUrl: string }>;
  verify(rawBody: string, signature: string): boolean;       // HMAC
  getRevShareReport(month: string): Promise<RevShareRow[]>;
}

// lib/tax/allocation.ts — משיכת מספר הקצאה בשם המרצה
export async function getAllocationNumber(sale: Sale, grant: TaxAuthGrant) {
  const res = await fetch(`${TAX_BASE}/Invoices/allocateNumber`, {
    method: "POST",
    headers: { Authorization: `Bearer ${grant.accessToken}` },
    body: JSON.stringify({ invoiceId: sale.id, amount: sale.amount, ... }),
  });
  if (res.status === 401) throw new TokenExpired(); // → retry אחרי refresh
  return (await res.json()).allocationNumber; // 9 ספרות
}

מנוע החשבוניות הפנימי

מה שמבדיל אותנו מאינטגרציה עם iCount/Green Invoice: אנחנו בונים את שכבת המסמך בעצמנו. ארבעה רכיבים שחייבים להיבנות נכון מההתחלה:

מספור רציף פר‑עוסק

לכל מרצה רצף עצמאי ורציף, ללא דילוגים. אטומי ב‑transaction (invoiceSeq++ תחת lock).

חתימה דיגיטלית

תעודת CA (Comsign) + סימון "מסמך ממוחשב" — דרישת סעיף 18(ב) להוראות ניהול ספרים.

קובץ אחיד (BKMVDATA)

ייצוא במבנה OPENFRMT לביקורת רשות המסים, פר‑עוסק. עמידות בפני שינוי (ללא מחיקה/עריכה).

ארכוב

שמירת PDF חתום ב‑Bunny, גיבוי 7 שנים, מסמך נעול לאחר הנפקה.

הערכת מאמץ: ה‑API למספר הקצאה פשוט. הכובד האמיתי — ניהול הרשאות פר‑מרצה + חידוש, חתימה דיגיטלית, ומבנה אחיד. דרך ביניים: חיבור ישיר לרשות המסים רק להקצאה, ומודול ייעודי לחתימה/מבנה — מקטין סיכון בלי לוותר על השליטה.

רגולציה ובקרות

  • לא מחזיקים כספים → אין רישיון "נותן שירותי תשלום", אין מאגד (חוק שירותי תשלום 2023). הכסף תמיד תלמיד→מרצה.
  • בקרת §50: חסימת הנפקת חשבונית מס לעוסק פטור/לא‑רשום. אימות שם↔מספר עוסק↔תיק לפני הנפקה.
  • הרשאה דיגיטלית פר‑מרצה + הסכם שירות בכתב (אחריות מס על המרצה, סעיף שיפוי).
  • דיווח נטו: אנחנו מדווחים רק על העמלה/ה‑rev‑share כהכנסה (מודל סוכן).
  • לפני השקה: חוות דעת עו"ד רגולציה + רו"ח, ושקילת Pre‑ruling מרשות המסים.
הסף לחובת מספר הקצאה יורד ל‑₪5,000 ב‑1.6.2026. רוב מכירות הקורסים הן B2C (תלמיד פרטי) → לא נדרש מספר הקצאה כלל; החובה רק במכירה לעוסק מעל הסף.
