מערכת דיוור — עורך מייל מבוסס בלוקים

פישוט פלואו יצירת המייל: מסך פתיחה (כותרת + משתנים + בחירת עורך ויזואלי/HTML) ואז עורך בלוקים inline — בדיוק אותו מודל בלוקים של הקורס: כפתור "הוסף בלוק תוכן" → block‑picker → בלוק נכנס inline ונערך במקום, עם toolbar מינימלי. אותו addBlock(), אותם .content-block — מה שמוריד תחזוקה ומאחד חוויה. סדרות מיילים מוגדרות בדף נפרד.

00סקירה + כללי CTO

העורך משרת שלושה סוגים: קמפיין בודד, שלב בסדרת drip, ומייל מבוסס‑טריגר — כולם משתמשים באותו עורך ובאותו מודל בלוקים.

5 כללי הזהב (חובה)

  1. React + Next.js — העורך הוא Client Component (אינטראקטיבי); הרשימות/הגדרות כ‑Server Components.
  2. TypeScript strict + Zod — סכמת הבלוקים מאומתת ב‑Zod (אותה סכמה בקורס ובמייל).
  3. Bunny CDN/Stream — תמונות הבלוקים ווידאו דרך Bunny; אין העלאות לשרת.
  4. Trigger.dev — כל שליחה, סדרה והשהיה — דרך Trigger.dev (בלי cron/setTimeout).
  5. Toast/Modal מעוצבים — אין alert/confirm; שמירה אוטומטית + טוסט.

01מה תוקן ב‑UX

לפני

  • פופאפ הגדרות ואז קפיצה לעורך נפרד
  • 5 טאבים תוכן (AI / HTML / ויזואלי / תצוגה / עורך HTML) — בלבול
  • אין באמת בונה‑בלוקים
  • אין ייצוג לסדרת מיילים / השהיות
  • סטטוסים לא ברורים (פעיל/מתוזמן/מתמשך)

אחרי

  • מסך פתיחה נקי: כותרת המייל + משתנים זמינים + בחירת עורך
  • בחירה מפורשת: עורך ויזואלי או בניית HTML
  • עורך בלוקים inline זהה לקורס — "הוסף בלוק" → block‑picker
  • עריכה במקום, toolbar מינימלי (מעלה/מטה/מחק) — בלי הגדרות מיותרות
  • סדרות מוגדרות בדף נפרד (לא בתוך העורך)
  • CTA ראשי יחיד (שמור) + שלח טסט + תצוגה מקדימה

עקרון מנחה (UI/UX Pro Max): פשוט ונקי — ברירות מחדל חזקות, מעט מאוד הגדרות לכל בלוק. עריכה inline (contenteditable) במקום פאנל מאפיינים. משתנים אישיים מוצגים בעברית (מה הם אומרים) ומוכנסים כטוקן.

03מודל הבלוקים המשותף

הלב של האיחוד: גוף המייל וגוף שיעור בקורס הם אותו מבנה — מערך בלוקים מסודר. רג'יסטרי אחד מגדיר כל סוג בלוק; הקשר (קורס/מייל) רק מסנן אילו בלוקים זמינים ואיך הם מרונדרים.

// lib/blocks/registry.ts — מקור אמת אחד
// 8 הסוגים הקיימים במערכת (זהים לקורס) — אותם labels/אייקונים
type BlockType =
  "h1" | "h2" | "text" | "quote"
  | "divider" | "cta" | "image" | "iframe";
  // תוספות ייעודיות-מייל (לוגו/רשתות) — אופציה עתידית, נכנסות לאותו picker

interface BlockDef<P> {
  type: BlockType;
  contexts: ("course" | "email")[];   // היכן זמין
  schema: ZodSchema<P>;                  // ולידציית props
  defaults: P;
  Editor: FC<{props:P; onChange:(p:P)=>void}>;  // תצוגה בקנבס
  Props:  FC<{props:P; onChange:(p:P)=>void}>;  // פאנל מאפיינים
  renderEmail?: (props:P, ctx:RenderCtx) => string; // HTML תואם-מייל
}

// גוף המסמך (מייל או שיעור) — אותו טיפוס
type BlockDoc = { id:string; type:BlockType; props:unknown }[];

תוצאה: שיפור בבלוק (נגישות, עיצוב) מגיע בו‑זמנית לקורס ולמייל. תוספת בלוק = רישום אחד ב‑registry.

04רינדור למייל + משתנים

בקנבס הבלוקים מרונדרים כ‑React (תצוגה חיה). לשליחה הם מתקמפלים ל‑HTML תואם‑מייל (טבלאות + CSS inline) — לקוחות מייל לא תומכים ב‑flex/grid. מומלץ MJML או רנדרר טבלאי פנימי.

// בעת שליחה לכל נמען — אינטרפולציה של משתנים
function compile(doc: BlockDoc, recipient: Recipient): string {
  const html = doc.map(b => registry[b.type].renderEmail!(b.props, ctx)).join("");
  return interpolate(html, {
    member_name: recipient.name,
    club_name:   ctx.club.name,
    login_url:   signedLoginLink(recipient),   // קישור חתום פר נמען
    club_slug:   ctx.club.slug,
  });
}
// {{member_name}} → "מיכל" ; חסר ערך → fallback מוגדר

תאימות מייל: CSS inline, רוחב ≤600px, טבלאות לפריסה, alt לכל תמונה, ו‑fallback לכל משתנה. בלוקים כמו columns מתקמפלים לטבלה, לא ל‑flex.

05מודל הנתונים (Prisma)

model EmailTemplate {
  id        String  @id @default(cuid())
  name      String          // שם פנימי
  subject   String
  preheader String?
  fromName  String
  doc       Json            // BlockDoc — מערך הבלוקים
  brand     Json            // רקע, צבע מותג, גופן, רוחב
  attachmentId String?
  contextType  String       // club | course | school
  contextId    String
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}

model Campaign {       // קמפיין בודד
  id         String @id @default(cuid())
  templateId String
  audience   Json           // everyone | leads | customers | tag:... | filter
  sendMode   SendMode       // NOW | SCHEDULED
  scheduledAt DateTime?
  status     CampaignStatus // DRAFT | SCHEDULED | SENDING | SENT
  stats      Json?          // נשלח/נפתח/קליקים
}

model EmailSeries {    // סדרת drip
  id      String @id @default(cuid())
  name    String
  trigger TriggerType?   // JOIN | CART_ABANDON | COMPLETE | INACTIVE | null(ידני)
  steps   EmailStep[]
  active  Boolean @default(false)
}
model EmailStep {
  id         String @id @default(cuid())
  seriesId   String
  templateId String
  order      Int
  delayHours Int            // השהיה לפני שלב זה
  condition  Json?          // אופציונלי: רק אם לא נפתח וכו'
}

// trigger/mailing/enroll-series.ts
export const enrollSeries = task({
  id: "enroll-series",
  run: async ({ seriesId, userId }) => {
    const steps = await getSteps(seriesId);
    let delay = 0;
    for (const s of steps) {
      delay += s.delayHours;
      await runSeriesStep.trigger({ stepId: s.id, userId }, { delay: `${delay}h` });
    }
  },
});

08רכיבי UI

מסך פתיחה + שלד

  • <EmailSetup> — כותרת + Preheader + משתנים
  • <EditorChoice> — ויזואלי / HTML
  • <EmailEditor> — top bar + קנבס + block‑picker
  • <VarsPopover> — משתנים בעברית → טוקן

בלוקים (משותף עם הקורס)

  • openBlockPicker() → .block-picker (8 פריטי .bp-item)
  • addBlock(type) — מכניס .content-block inline
  • .block-toolbar — מעלה/מטה/מחק בלבד
  • עריכה contenteditable במקום — ללא PropertiesPanel

נגישות: הוספה בלחיצה מה‑block‑picker (לא תלוי גרירה), הזזה דרך toolbar (מעלה/מטה), aria-labels על כל כפתורי הבלוק, focus עובר אוטומטית לבלוק החדש, ו‑prefers-reduced-motion לאנימציות.
