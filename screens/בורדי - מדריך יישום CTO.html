ארבע החלטות המוצר

Hub, לא נתב

המרצה בוחר את הבורדי הרלוונטי מראש. אין שלב סיווג‑כוונה → אפס latency‑נתב, אפס טעויות ניתוב.

מעטפת אחת

ליבה משותפת (Runtime); לכל בורדי סקילים, scope דאטא וכלים משלו. גישה A — מודל אחד + כלים.

בורדי פועל

בונה‑הקורס משנה את המערכת בפועל: ליי‑אאוט, פרקים, שיעורים, קבצים, הערות. דרך Action API.

Thread לפי פרויקט

שיחה נשמרת וקשורה לישות (קורס/קמפיין) — אפשר לחזור ולהמשיך לבנות אותו קורס מחר.

MVP: צד המרצה. בורדי‑דגל = בונה הקורסים (פעיל), אחריו יוצר הדיוור. מנתח תלמידים ו‑עדכון קורס קיים נכנסים מיד אחרי על אותה מעטפת. בורדי‑תלמיד (RAG על תמלולי שיעורים) = שלב 2.

מודל ה‑Hub

מסך הבית של בורדי מציג כרטיסי משימה ("מה תרצה לעשות היום?"). המרצה בוחר → נכנס לבורדי ממוקד. אין ניתוב אוטומטי — הבחירה היא ה"ניתוב".

לולאת ה‑Orchestrator

הלב של גישה A: מודל אחד עם כלים. הבורדי הנבחר קובע את ה‑system‑prompt ואת סט הכלים; הלולאה רצה עד שהמודל מסיים או מגיע לגבול צעדים.

// lib/boardi/run.ts — לולאת ריצה אחת של בורדי
export async function runBoardi(threadId: string, userMsg: string) {
  const thread = await loadThread(threadId);          // כולל tenancy
  const boardi = boardiRegistry[thread.boardiType];  // prompt + tools + scope
  const context = await rag.retrieve(boardi.scope, thread, userMsg);

  let messages = [systemPrompt(boardi, context), ...thread.history, user(userMsg)];
  for (let step = 0; step < MAX_STEPS; step++) {
    const res = await llm.chat({ messages, tools: boardi.tools }); // Adapter
    if (!res.toolCalls?.length) return finalize(thread, res);   // תשובה סופית

    for (const call of res.toolCalls) {
      if (tools[call.name].kind === "write")
        await requireConfirmation(thread, call);  // preview → אישור מרצה
      const out = await executor.run(call, thread.teacherId);
      messages.push(toolResult(call, out));
    }
  }
}
כלי write אינו מבוצע בזרימה הרגילה — הוא מוחזר ל‑UI כ‑preview, המרצה מאשר, ורק אז ה‑executor מריץ אותו. פעולות ארוכות (תמונות/מצגות) נדחפות ל‑Trigger.dev ומעדכנות את ה‑thread כשמסתיימות.

RAG ומקורות דאטא

מקורות נתונים וחיבורים

האנליטיקס של בורדי מבוסס על מודל דאטא מעשי: בורדי לא מחווט APIs חיצוניים בכל בקשה, והמרצה לא מחבר עשרות אינטגרציות. בורדי קורא משכבת Metrics Layer פנימית ומאוחדת ש‑WHITEBOARD ממלאת. כלי השליפה של מנתח‑התלמידים (query_analytics) מדבר רק מול החוזה הפנימי הזה — לעולם לא מול Meta/Google בזמן שאילתה.

העיקרון: שכבת סנכרון (קונקטורים native + Make.com — כבר בסטאק של WHITEBOARD) מושכת את המקורות החיצוניים בתזמון אל Metrics Store פנימי. בורדי קורא מה‑Store בלבד → אפס קריאות API בזמן שאילתה, ביצועים יציבים, ואין תלות בזמינות ספק חיצוני בזמן אמת. זה מה שהופך את זה למעשי.

חיבור מטא — איך זה עובד בפועל

מטא הוא לא "כפתור אחד". מאחוריו זרימת Facebook Login for Business על אפליקציית מטא של WHITEBOARD. חיבור אחד פותח את כל נכסי מטא — עמוד פייסבוק, אינסטגרם, חשבון מודעות ו‑Pixel — ולכן במסך החיבורים זה כרטיס אחד ("מטא · Facebook & Instagram"), לא ארבעה.

מימה קורה
WHITEBOARD · פעם אחתרישום Meta App עם Business Login configuration, ומעבר App Review של מטא לקבלת גישה מתקדמת ל‑scopes. תהליך פיתוח חד‑פעמי — לא של הלקוח.
המרצה · 1לוחץ "חבר" → נפתחת חלונית מאובטחת של Facebook ומתחבר לחשבון שלו.
המרצה · 2בחירת נכסים — מטא מציגה אילו נכסים לחשוף: עמוד, אינסטגרם, חשבון מודעות, Pixel/Dataset. אפשר לבחור חלק.
המרצה · 3אישור הרשאות (scopes, קריאה בלבד) → חוזרים למערכת.
WHITEBOARD · מאחורי הקלעיםshort‑lived token → החלפה ל‑long‑lived / System‑User token דרך ה‑Business Manager; נשמר מוצפן ומרוענן אוטומטית. sync-integrations מושך דרך Graph API / Marketing API אל ה‑Metrics Store.

הטמעה — מדריך מודרך ומחולל קודים (MVP)

שני מודלים: (A) Platform App — WHITEBOARD כאפליקציית מטא/גוגל אחת שעברה App Review, המרצה רק עושה OAuth; (B) Bring‑Your‑Own App — המרצה יוצר אפליקציה משלו ומדביק App ID/Secret, ו‑WHITEBOARD מייצר עבורו את הערכים להחזיר לקונסולה. ל‑MVP יוצאים עם B (לא תלוי ב‑App Review) ומהגרים ל‑A מאוחר יותר — החוזה הפנימי לא משתנה. מסך "מדריך חיבורים — מטא וגוגל" מפרק כל שלב עם צילום מסך מדומה, ריחוף‑הסבר, שדות למילוי ומחולל קודים. הערכים שהמערכת מייצרת והמרצה מדביק אצל הספק:

רמת הוודאות — Course Understanding Meter

כשבונה‑הקורס עובד עם מרצה, הוא שואל שאלות עד שהוא מבין את הקורס והמטרות. "רמת הוודאות" היא מדד חי שעולה והופך לירוק ככל שההבנה מתגבשת — ומשמש שער: בורדי לא בונה ליי‑אאוט מתחת לסף.

סטטוס: את סקיל השאלות (ניהול האינטייק החכם) נבנה בהמשך — הוא לא ב‑MVP הראשון. מה שמאופיין כאן ומעוצב עכשיו הוא מנגנון רמת הוודאות וה‑UI שלו, כדי שיהיה מוכן כשנחבר את הסקיל.

איך הוודאות מחושבת

הוודאות נגזרת מ‑שדות‑ליבה שבורדי צריך למלא בביטחון כדי לתאר קורס. כל שדה שנתפס בביטחון מעלה את הציון ומדליק "ממד" ירוק ברייל:

ממד (core field)מה נלכדמשקל
domain · תחום הקורסנושא/תחום התוכן~15%
audience · קהל יעדרמה (מתחילים/מקצוענים) + הקשר~20%
goal · מטרת הבוגרהתוצאה שהבוגר משיג~25%
scope · היקף ותכולהמס' פרקים/שיעורים, עומק~20%
style · סגנון למידהתיאוריה+תרגול / פרויקט / מיקרו~20%
 
// lib/boardi/certainty.ts — ציון וודאות + שער בנייה
const WEIGHTS = { domain: 15, audience: 20, goal: 25, scope: 20, style: 20 };
const BUILD_THRESHOLD = 85; // מתחת לזה — ממשיכים לשאול

export function certainty(fields: Record<string, { value?: string; confidence: number }>) {
  let score = 0;
  for (const [k, w] of Object.entries(WEIGHTS))
    score += (fields[k]?.confidence ?? 0) * w;   // confidence 0..1
  return Math.round(score);
}
export const canBuildLayout = (s: number) => s >= BUILD_THRESHOLD;
הערכת confidence: אחרי כל תשובה, בורדי (ה‑LLM) מחזיר לכל שדה value + confidence 0..1 כפלט מובנה (StructuredOutput). ה‑UI ממפה את הציון לצבע (ורוד→צהוב→ירוק) ולגרף העולה, ומדליק את הממד המתאים.

עיצוב (UI) — לפי מערכת WHITEBOARD

  • רייל ימני "רמת הבנת הקורס": מד אחוזים גדול (Fredoka, tabular‑nums) + גרף שטח עולה שצבעו מתחלף ורוד→צהוב→ירוק, וצ'קליסט 5 הממדים שמתמלאים ירוק.
  • תוויות מצב: "מתחיל להכיר…" → "אוסף את התמונה" → "מבין את הכיוון" → "מוכן לבנות ליי‑אאוט 🎉".
  • שער חזותי: כפתור "בנה ליי‑אאוט" מושבת עד חציית הסף; הרייל מסביר למרצה מה עוד חסר.
  • כל הטוקנים במערכת WHITEBOARD (navy/cyan/pink/green, Heebo+Fredoka, RTL, מצב כהה). ראה מסכים מחודשים/בורדי/.

ניהול צ'אטים — Thread לפי פרויקט

שיחה נשמרת ב‑DB ומאורגנת לפי פרויקט (ישות יעד), לא רק לפי בורדי. כך המרצה חוזר וממשיך לבנות את אותו קורס. תוצרים (ליי‑אאוט, מייל) נשמרים כ‑artifacts שמצביעים לישות שנוצרה במערכת.

model BoardiThread {

  targetId    String?               // הישות שהשיחה עוסקת בה

  messages    BoardiMessage[]
  artifacts   BoardiArtifact[]
  createdAt   DateTime @default(now())
  updatedAt   DateTime @updatedAt
}

ניהול context window: כשה‑thread גדל, הודעות ישנות עוברות summarization ל‑memory תמציתי; artifacts נשמרים כהפניות (refId) ולא כטקסט גולמי בפרומפט. בכל בורדי מוצגת רשימת "שיחות אחרונות" → "המשך מאיפה שהפסקת".

DATA Board — לוח הנתונים החכם

ברייל האנליטיקס אין פאנל מסננים — יש DATA Board: לוח נעיצה של דוחות חכמים. כל כרטיס נתונים או תובנה שבורדי שולח בצ'אט נושא כפתור נעיצה (📌); לחיצה נועצת אותו ל‑DATA Board עם חותמת תאריך/שעה. המודל המנטלי: המרצה הוא כמו מנכ"ל שמשוחח עם אנליסט ומרכיב לעצמו דוח סיכום חכם ומעוצב מראש.

הורד דוח מעוצב: כפתור ב‑DATA Board מרכיב דוח HTML עצמאי ואלגנטי מכל הפריטים הנעוצים — במותג WHITEBOARD, עם כותרת navy (לוגו + "דוח DATA Board · בורדי" + תאריך), כל גרף/תובנה שנעצה מרונדרת, וסיומת. ניתן להדפסה ל‑PDF. הרכבה = compose צד‑לקוח של ה‑artifacts הנעוצים, או job generate_report (Trigger.dev) לרינדור PDF בשרת. הדוח משתמש מחדש במערכת העיצוב של המותג.

ארבעת עמודי האנליטיקס

מנתח התלמידים מאורגן בארבעה תחומי‑על, וכל אחד מסתעף למדדים פנימיים; כל ניתוח מסתיים בהמלצה של בורדי (insight) ויכול לבצע handoff (למשל ליוצר‑הדיוור לקמפיין החזרה):

  • סקירה כללית (Overview) — סה"כ תלמידים, % פעילים, התקדמות ממוצעת, שביעות רצון; דונאט סטטוס; גרף מגמת מעורבות.
  • שיווק (Marketing) — חשיפות, CTR, המרה לליד, עלות‑לליד; משפך שיווקי (חשיפות→קליקים→לידים→הרשמות); מקורות תנועה.
  • מכירות (Sales) — הכנסה, מס' מכירות, AOV, החזרים; מגמת הכנסה; קורסים נמכרים ביותר.
  • שירות · איכות הקורסים (Service) — שיעור השלמה, דירוג ממוצע, NPS, שאלות פתוחות; משפך התקדמות/נטישה; תלמידים בסיכון.

Handoff בין‑בורדי

בורדי שסיים משימה מציע מעבר לבורדי אחר עם הקשר טעון. לחיצה פותחת thread חדש בבורדי‑היעד שבו ה‑context כבר מוזרק כהודעת מערכת ראשונה.

1
בונה‑הקורס מסיים ליי‑אאוט → מציג הצעה: "רוצה שאכין מייל השקה לקורס?"
2
המרצה לוחץ → נוצר HandoffRequest עם { from, to: "MAILING", context: { targetType: "COURSE", targetId, summary } }.
3
נפתח thread חדש ביוצר‑הדיוור; ה‑context מוזרק כ‑system message → הבורדי כבר "מכיר" את הקורס.
דוגמאות נוספות: מנתח‑תלמידים → יוצר‑דיוור (קמפיין החזרת נוטשים) · בונה‑קורס → יצירת‑תמונות (קאבר) · מנתח‑תלמידים → עדכון‑קורס (פיצול שיעור עם נטישה גבוהה).

1

2

מיפוי בונה‑הבלוקים: יוצר‑הדיוור מייצר את אותו מבנה בלוקים שהעורך הקיים משתמש בו (JSON blocks), כך שהתוצר ניתן לעריכה ידנית אחרי הבורדי. פורמט הבלוקים = פער פתוח למיפוי מול הצוות.

גבולות ובטיחות

  • אישור לכל כתיבה — כלי write מחזיר preview; ביצוע רק אחרי אישור מרצה מפורש.
  • Tenancy מוחלט — כל שליפה/כתיבה מסוננת ב‑teacherId (RLS + ctx בכל כלי). בורדי לא ניגש לדאטא של מרצה אחר.
  • הרשאות לפי מסלול — יכולות בורדי נגזרות ממסלול המרצה, נאכף בשרת (לא רק ב‑UI); ראה טבלת הגייטינג למטה.
  • Rate / cost guardrails — תקרות על קריאות LLM, מחקר ויצירת תמונות פר‑מרצה/פר‑חודש, נגזרות מהמסלול.
  • שקיפות + Audit log — בורדי מציג אילו פעולות ביצע ואילו מקורות שלף; כל כתיבה נרשמת ל‑boardi_audit (teacherId, tool, entity, before/after, ts).

מסך דמו שני, בנוי סביב 4 עמודי אנליטיקס (סקירה כללית · שיווק · מכירות · שירות ואיכות), שכל אחד מסתעף למדדים פנימיים: כרטיסי KPI עם דלתות, דונאט פילוח סטטוס, גרף מגמה רב-סדרתי, עמודות לפי קורס, משפך התקדמות עם נקודת נטישה מודגשת, רשימת תלמידים בסיכון, והמלצות פעולה. במקום מסננים — רייל "DATA Board": לוח נתונים חכם שאליו נועצים כל כרטיס/תובנה שבורדי שולח (כפתור 📌 + חותמת זמן), עם כפתור "הורד דוח מעוצב" שמרכיב דוח HTML עצמאי במותג הקורס. הרייל כולל גם פיד יזום "בורדי ממליץ לנעוץ" — התראות חיות (סיכון, נטישה, אבני דרך) שכל לחיצה עליו פותחת אותן בשיחה (הצ'אט הוא ה-scope, לא פילטר גלובלי). Handoff לקמפיין החזרה.

הקנבס הפעיל (Active Canvas)

במקום שהצ'אט יתאר מה יעשה — בורדי עושה מול העיניים של המרצה והוא מאשר. הצ'אט נעגן לצד שמאל כחלון צף נפרד, והאזור המרכזי הוא מסך המערכת. בורדי מנווט לעמוד הנכון, מדגיש את השדה, ממלא ערך ממתין (צהוב), מציג preview, ורק אחרי אישור הערך נטמע (הבזק ירוק). שינוי הגדרות, בניית מייל בבלוקים, הוספת תלמיד לקורס, שינוי צבעי מותג — הכל דרך אותו חלון.

קריטי — הקנבס חייב לרוץ על המסכים האמיתיים שלנו. מסך הדמו (בורדי - קנבס פעיל.html) בונה מסכי הגדרות/דיוור/תלמידים מדומים כדי להמחיש את האינטראקציה בלבד — אלה אינם מסכי WHITEBOARD האמיתיים. במימוש בפועל הקנבס מרנדר את מסכי הפרודקשן הקיימים של WHITEBOARD (מסך ההגדרות האמיתי, עורך הדיוור האמיתי, מסך התלמידים האמיתי), ובורדי פועל עליהם — לא על שכפול. אין לבנות מסכים חדשים לקנבס; יש לחבר אליו את הקיימים.

איך זה מתחבר למסכים האמיתיים (שכבת UI-Driver):

שכבהתפקיד
Canvas Hostעוטף את מסכי המערכת הקיימים (route/רכיב React אמיתי) בתוך אזור הקנבס — אותו קוד שהמרצה רואה בניווט הרגיל, לא רכיב מוקאפ.
UI-Driverגשר בין פקודות בורדי לרכיבי המסך: navigate(screen), highlight(field), setPending(field,value), flash(field). מזהה שדות דרך data-boardi-field על הרכיבים האמיתיים — מוסיפים תגיות זיהוי למסכים הקיימים, לא בונים חדשים.
Action APIהאישור מפעיל בדיוק את אותו wbApi שהמסך עצמו קורא לו בעריכה ידנית — כך הוולידציה, ההרשאות והתנאים זהים בין "בורדי מילא" ל"המרצה מילא".
הקנבס משנה את חוויית הביצוע — לא את חוזה הבטיחות. כל פעולה עדיין עוברת preview→confirm + guardrails (tenancy, הרשאות מסלול, cost caps). מסך הדמו: מסכים מחודשים/בורדי/בורדי - קנבס פעיל.html (להמחשת אינטראקציה בלבד).

פערים פתוחים להחלטה

שלושה נושאים שטרם הוכרעו וצריכים החלטה לפני מימוש הקנבס:
  • Undo / rollback. יש preview→confirm, אבל אין עדיין "בטל פעולה אחרונה" אחרי שהערך נטמע. בעולם שבו בורדי כותב לשדות אמיתיים — צריך undo לכל כתיבה (snapshot של before מה‑audit log, או פעולת reverse ב‑wbApi).
  • כשל אחרי אישור. מה קורה כש‑wbApi נכשל אחרי שהמרצה אישר (ולידציה, הרשאת מסלול, קונפליקט עריכה בו‑זמנית)? הקנבס צריך להחזיר את השדה למצב קודם ולהציג שגיאה ברורה — לא להשאיר ערך "ממתין" תקוע. (קשור ל‑optimistic locking אם המרצה עורך במקביל.)
  • מובייל. "צ'אט‑שמאל + מסך‑מערכת" הוא דפוס דסקטופ. במסך צר צריך להחליט: צ'אט מלא‑מסך עם מעבר לקנבס, או קנבס עם צ'אט מגירה. לא מוגדר עדיין.
