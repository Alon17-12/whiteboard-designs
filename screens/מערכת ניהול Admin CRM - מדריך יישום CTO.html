מערכת ניהול Admin/CRM
מדריך יישום טכני

אזור admin/ בתוך פלטפורמת WHITEBOARD — לא מערכת נפרדת. יושב על אותו DB, אותו stack ואותם משתמשים, ומספק להנהלה תצוגת בקרה אחת לשיווק, מכירות, שירות ומוצר. המסמך מגדיר ארכיטקטורה, מודל נתונים, עבודות רקע, הרשאות ושלביות.

00סקירה + כללי CTO

7 מודולים + דשבורד ראשי, כולם נשענים על דאטה שכבר קיימת. שכבת אגרגציה (Trigger.dev) מחשבת KPIs מתוזמן, והממשק מציג דשבורדים בזמן אמת עם התראות פעולה.

5 כללי הזהב (חובה)

  1. React + Next.js — App Router, Server Components כברירת מחדל; Client רק לאינטראקטיביות (גרפים, צ'אט).
  2. TypeScript strict + Zod — ללא any; ולידציה בכל קלט.
  3. Bunny CDN/Stream — כל מדיה דרך Bunny.
  4. Trigger.dev — כל עבודת רקע (חישוב KPI, התראות, דוחות, תזכורות) — בלי cron ידני / setTimeout.
  5. Toast/Modal מעוצבים בלבד — אין alert/confirm/prompt.

שיווק ומשפך

לידים, ניסיון, נטישת עגלה, המרה לפי מקור.

לקוחות ומנויים

מסלולים, MRR, נטישה, בריאות חשבון.

אנליטיקת מרצים

מכירות Enterprise

pipeline, הצעות מחיר, בקשות, חוזים.

שותפים ועמלות

מסלולים, הפניות, עמלות אוטומטיות.

שירות (CS)

01ארכיטקטורה

המודול הוא route group ב‑App Router עם middleware להרשאות צוות. שכבת קריאה (read‑model) מחושבת מראש כדי שדשבורדים יטענו מהר, במקום aggregation כבד בזמן אמת.

// מבנה תיקיות
app/
  (admin)/                 // route group, מוגן ב-middleware
    layout.tsx             // shell: sidebar + topbar + theme
    dashboard/page.tsx     // מרכז הבקרה
    marketing/page.tsx
    customers/page.tsx
    lecturers/page.tsx
    enterprise/page.tsx
    affiliates/page.tsx
    service/page.tsx
    product/page.tsx
    billing/page.tsx       // סליקה (עתידי)
  api/admin/
    kpis/route.ts          // קריאת read-model
lib/admin/
  rbac.ts                  // בקרת גישה לפי תפקיד
  kpi.ts                   // חישובי KPI
trigger/admin/             // עבודות רקע

שכבות

  • מקורות דאטה: ה‑DB הקיים (תלמידים, קורסים, מנויים, צפיות) · נתוני סליקה/חשבוניות · שותפים והפניות · אירועי שיווק (לידים, ניסיונות, נטישות).
  • אגרגציה: עבודות Trigger.dev מתוזמנות כותבות ל‑KpiSnapshot (read‑model) — הדשבורד קורא ממנו, לא מחשב חי.
  • הצגה: Server Components טוענים מה‑read‑model; גרפים/אינטראקציה ב‑Client.
  • התראות: זיהוי at‑risk / נוטשי עגלה / עמלות לתשלום → פריטי פעולה בדשבורד + מייל/התראה.

עיקרון: אף KPI לא מחושב בזמן בקשה. הכול מ‑read‑model מעודכן, כדי שטעינת דשבורד תהיה מתחת ל‑200ms גם על מיליוני שורות.

03מכירות מול תזרים

ההבחנה הקריטית: מכירות (הכרה) מול תזרים (כסף בפועל). לקוח ששילם שנה מראש נכנס במלואו לתזרים החודש, אך נזקף 1/12 בלבד למכירות בכל חודש.

מושגמודלמתי נרשם
תזרים (Cash)CashTransactionבמלואו ביום החיוב בפועל
מכירות / MRR (Recognition)RevenueRecognitionמתפרס: שנתי → 1/12 לחודש
 
model CashTransaction {
  id          String @id @default(cuid())
  customerId  String
  amount      Int            // אגורות — נכנס במלואו
  paidAt      DateTime
  billingType BillingType   // ONE_TIME | MONTHLY | ANNUAL_PREPAID
}
model RevenueRecognition {
  id          String @id @default(cuid())
  txId        String
  periodMonth String          // "2026-06"
  amount      Int            // חלק שנזקף לחודש זה
}

// בעת קליטת תשלום: רושמים תזרים מלא + יוצרים זקיפות הכרה
function recognize(tx: CashTransaction) {
  if (tx.billingType === 'ANNUAL_PREPAID')
    return spread(tx.amount, 12);  // 12 זקיפות חודשיות
  return [{ periodMonth: month(tx.paidAt), amount: tx.amount }];
}

בדשבורד: MRR = sum(RevenueRecognition where periodMonth=current); סה״כ הכנסה החודש (תזרים) = sum(CashTransaction where month(paidAt)=current). הגרף מצייר את שני הקווים זה לצד זה.

04עבודות רקע (Trigger.dev)

כל החישוב הכבד וההתראות — ב‑Trigger.dev. שמות: trigger/admin/[action].ts.

Jobתזמוןתפקיד
compute-kpisכל שעהמעדכן KpiSnapshot (MRR, תזרים, churn, לידים, פר מרצה).
recognize-revenueיומייוצר זקיפות הכרה לתשלומים חדשים (פריסת שנתי).
detect-at-riskיומימסמן לקוחות/מרצים בסיכון נטישה לפי ירידת שימוש.

weekly-digestראשון 09:00מייל סיכום שבועי להנהלה.
sync-clearing עתידיwebhookקולט עסקאות מחברת הסליקה ומפעיל הפקת חשבונית.
 
// trigger/admin/compute-kpis.ts
export const computeKpis = schedules.task({
  id: "compute-kpis",
  cron: "0 * * * *",
  run: async () => {
    const mrr  = await sumRecognition(currentMonth());
    const cash = await sumCash(currentMonth());
    await upsertSnapshot([
      { key: "mrr",  scope: "global", value: mrr },
      { key: "cash", scope: "global", value: cash },
    ]);
  },
});

05סליקה ותשלומים עתידי

תשתית בלבד כרגע. מתחבר ליוזמת הסליקה: כל מרצה מסוף עצמאי, הכסף עובר ישירות תלמיד→מרצה דרך חברת הסליקה, WHITEBOARD מפיקה חשבוניות בשם המרצה וגובה rev‑share.

model ClearingTransaction {
  id          String @id @default(cuid())
  lecturerId  String
  studentId   String
  amount      Int
  billingType BillingType
  invoiceId   String?        // חשבונית שהופקה ע"י WB
  wbCommission Int           // rev-share מחברת הסליקה
  providerRef String          // מזהה אצל ספק הסליקה
}

תלוי יוזמת הסליקה: הפקת החשבוניות, ה‑rev‑share וההתחברות לרשות המסים מפורטים במדריך ה‑CTO של "סליקה וחשבוניות". מודול זה צורך את הנתונים בלבד.

06הרשאות ותפקידים

07API ואינטגרציות

  • מקורות שיווק: UTM/מקור נשמר על Lead.source מטופס ההרשמה; אינטגרציה ל‑Google/Meta Ads לייחוס המרות (v2).
  • חיוב והכרה: webhooks מספק הסליקה → CashTransaction → recognize-revenue.
  • חשבוניות: דרך מנוע החשבוניות של WHITEBOARD (יוזמת הסליקה) → רשות המסים Open API.
  • מייל/התראות: Trigger.dev → ספק מייל; תזכורות נטישה, digest שבועי, התראות at‑risk.
  • API פנימי: GET /api/admin/kpis?scope=global מחזיר read‑model; Zod על כל קלט/פלט.
